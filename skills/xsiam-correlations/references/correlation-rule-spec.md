# Correlation Rule YAML Specification

This document defines the complete YAML schema for Cortex XSIAM correlation rules.
Correlation rules wrap XQL detection logic with alert metadata, severity, MITRE ATT&CK
mapping, and suppression configuration.

---

## Canonical Field Order

Fields must appear in this exact order to match XSIAM export format. Omitting optional
fields is acceptable; reordering is not.

```
global_rule_id
rule_id
rule_name
rule_labels
rule_source
name
alert_name
description
dataset
action
alert_domain
alert_category
execution_mode
search_window
crontab
xql_query
severity
user_defined_severity
user_defined_category
suppression_enabled
suppression_duration
suppression_fields
mitre_defs
mapping_strategy
alert_fields
drilldown_query_timeframe
investigation_query_link
```

## Field Reference Table

| Field | Type | Required | Valid Values | Description |
|-------|------|----------|-------------|-------------|
| `global_rule_id` | string | **Yes** | UUID v4 (lowercase with hyphens) | Globally unique identifier for the rule. Generate a fresh UUID for each new rule. |
| `rule_id` | string | **Yes** | snake_case identifier | Machine-readable rule ID. Convention: `<vendor>_<detection_name>` or `custom_<detection_name>`. |
| `rule_name` | string | **Yes** | Human-readable name | Display name shown in the XSIAM rules list. |
| `rule_labels` | list | Optional | List of strings | Tags for organizing rules (e.g., `["brute_force", "credential_access"]`). |
| `rule_source` | string | Optional | Freeform | Origin of the rule (e.g., `xsiam-buddy`, `SOC-team`, vendor name). |
| `name` | string | **Yes** | Human-readable name | Alert name template. **Must match `alert_name` exactly.** |
| `alert_name` | string | **Yes** | Human-readable name | Alert name displayed in the alerts table. **Must match `name` exactly.** |
| `description` | string | **Yes** | Freeform | Detailed description of what the rule detects and why it matters. |
| `dataset` | string | **Yes** | `alerts` | Always `alerts` for correlation rules. |
| `action` | string | **Yes** | `ALERTS` | Always `ALERTS` -- correlation rules produce alerts. |
| `alert_domain` | string | Optional | Freeform | Domain classification (e.g., `NETWORK`, `IDENTITY`, `ENDPOINT`, `CLOUD`). |
| `alert_category` | string | Optional | Freeform | Category classification (e.g., `Credential Access`, `Lateral Movement`). |
| `execution_mode` | string | **Yes** | `REAL_TIME`, `SCHEDULED` | How the rule is triggered. See Execution Mode Guide below. |
| `search_window` | string | Conditional | Duration string (e.g., `1m`, `15m`, `1h`, `24h`) | Lookback window for the query. Required for `SCHEDULED`; optional for `REAL_TIME`. |
| `crontab` | string | Conditional | Standard cron expression | Schedule for `SCHEDULED` rules (e.g., `*/5 * * * *`). Required when `execution_mode: SCHEDULED`. Not used for `REAL_TIME`. |
| `xql_query` | string | **Yes** | XQL query (literal block scalar `\|`) | The detection logic. Must be valid XQL. Embedded as a YAML literal block scalar. |
| `severity` | string | **Yes** | Severity enum (see below) | Static severity assigned to alerts generated by this rule. |
| `user_defined_severity` | object | Optional | Mapping of field conditions to severity overrides | Allows dynamic severity override based on field values at alert time. |
| `user_defined_category` | string | Optional | Freeform | Custom category visible in the alerts table. |
| `suppression_enabled` | boolean | **Yes** | `true`, `false` | Whether duplicate alert suppression is active. |
| `suppression_duration` | string | Conditional | Duration string (e.g., `30m`, `1h`, `24h`) | How long to suppress duplicate alerts. Required when `suppression_enabled: true`. |
| `suppression_fields` | list | Conditional | List of field name strings | Fields that define a "duplicate" alert. Required when `suppression_enabled: true`. |
| `mitre_defs` | object | **Yes** | Technique-to-tactics mapping | MITRE ATT&CK mapping. See MITRE ATT&CK Mapping section below. |
| `mapping_strategy` | string | Optional | `XQL`, `CUSTOM` | How alert fields are populated. Default: `XQL`. |
| `alert_fields` | list | Optional | List of field mapping objects | Custom field mappings for the alert. Used when `mapping_strategy: CUSTOM`. |
| `drilldown_query_timeframe` | string | Optional | Duration string | Time window for the drilldown query link in the alert view. |
| `investigation_query_link` | string | Optional | URL or query reference | Link to an investigation query for deeper analysis. |

## Severity Guide

### Enum Values

XSIAM uses a fixed set of severity enum values. There is **no SEV_050** -- this is
intentional and not a gap.

| Enum Value | Meaning | Use When |
|-----------|---------|----------|
| `SEV_010_CRITICAL` | Critical | Confirmed active compromise, data breach, or ransomware execution |
| `SEV_020_HIGH` | High | Strong indicator of compromise requiring immediate investigation |
| `SEV_030_MEDIUM` | Medium | Suspicious activity warranting investigation within SLA |
| `SEV_040_LOW` | Low | Anomalous but potentially benign activity for triage |
| `SEV_060_INFORMATIONAL` | Informational | Audit events, policy violations, or telemetry for enrichment |

### Static vs Dynamic Severity

**Static severity** (set in YAML `severity` field):
- Use when the detection always warrants the same response priority
- Example: honeypot/canary alerts are always HIGH because any trigger is suspicious

**Dynamic severity** (computed in XQL via `if()`):
- Use ONLY when the source data has a numeric score but no discrete severity field
- Compute `xdm.alert.severity` in the XQL query using an `if()` chain
- Always set a static fallback severity in the YAML `severity` field
- Example: Darktrace `percentscore` mapped to CRITICAL/HIGH/MEDIUM/LOW

**Rule of thumb:** If the source data already has a severity/priority field with discrete
values (e.g., "critical", "high", "medium"), map it directly in the YAML -- do NOT use
dynamic severity in XQL.

## Execution Mode Guide

### REAL_TIME

Triggered on each matching event as it is ingested. Best for single-event detections
where every match independently warrants an alert.

**Characteristics:**
- No `crontab` field (omit it entirely)
- `search_window` is optional; if set, keep it short (e.g., `1m`)
- Lower latency -- alerts fire within seconds of event ingestion
- Higher alert volume potential -- every matching event produces an alert

**Best for:**
- Canary/honeypot triggers
- Known-bad IOC matches
- Critical policy violations (e.g., disabled MFA login)
- Firewall deny events on sensitive ports

### SCHEDULED

Runs on a cron schedule. Best for aggregation and threshold detections that require
counting or grouping events over a time window.

**Characteristics:**
- `crontab` field required (standard 5-field cron expression)
- `search_window` required -- defines the lookback period for the query
- `search_window` should be at least 2x the cron interval to avoid gaps
- Higher latency -- alerts fire only at scheduled intervals

**Best for:**
- Brute force / password spray (count failures over time)
- Data exfiltration volume thresholds
- Lateral movement (distinct host count over time)
- Anomaly detection with baseline comparison

**Common crontab patterns:**

| Pattern | Meaning |
|---------|---------|
| `*/5 * * * *` | Every 5 minutes |
| `*/15 * * * *` | Every 15 minutes |
| `0 * * * *` | Every hour |
| `0 */4 * * *` | Every 4 hours |
| `0 0 * * *` | Daily at midnight |

## Suppression Patterns

Suppression prevents duplicate alerts from flooding the SOC. Choose suppression fields
that define what makes two alerts "the same incident."

| Detection Type | Suppress By | Recommended Duration | Rationale |
|---------------|-------------|---------------------|-----------|
| Brute force / password spray | `source_ip` | 1h | Same source attacking = same incident |
| Lateral movement | `hostname` | 2h | Same compromised host = same incident |
| Data exfiltration | `user_name` | 4h | Same user exfiltrating = same incident |
| Malware detection | `file_hash` | 24h | Same malware = same incident |
| Repeated failed auth | `user_name`, `source_ip` | 30m | Same user from same source = same incident |
| Canary / honeypot trigger | `alert_name` | 1h | Same canary = same incident |
| Firewall block (repeated) | `source_ip`, `destination_port` | 1h | Same source hitting same port = same incident |
| Privilege escalation | `user_name` | 2h | Same user escalating = same incident |
| Suspicious process execution | `hostname`, `process_name` | 1h | Same process on same host = same incident |

**Guidelines:**
- Always enable suppression unless the detection is extremely rare (< 1 alert/day expected)
- Shorter durations for time-sensitive detections (auth failures)
- Longer durations for persistent threats (malware, exfiltration)
- Use multiple suppression fields when a single field is too broad

## MITRE ATT&CK Mapping

### Structure

The `mitre_defs` field maps MITRE ATT&CK technique IDs to their associated tactic IDs:

```yaml
mitre_defs:
  "T1110":
    - "TA0006"
  "T1078":
    - "TA0001"
    - "TA0003"
```

### Format Rules

- Technique IDs: `T` followed by 4 digits (e.g., `T1110`) or with sub-technique
  (e.g., `T1110.003`)
- Tactic IDs: `TA` followed by 4 digits (e.g., `TA0006`)
- A single technique can map to multiple tactics
- A rule can map to multiple techniques

### Common Mappings by Detection Category

| Detection Category | Technique | Tactic(s) |
|-------------------|-----------|-----------|
| Brute force | T1110 | TA0006 (Credential Access) |
| Password spray | T1110.003 | TA0006 (Credential Access) |
| Valid accounts | T1078 | TA0001 (Initial Access), TA0003 (Persistence) |
| MFA bypass | T1556.006 | TA0006 (Credential Access), TA0005 (Defense Evasion) |
| Lateral movement (RDP) | T1021.001 | TA0008 (Lateral Movement) |
| Lateral movement (SMB) | T1021.002 | TA0008 (Lateral Movement) |
| Data exfiltration (network) | T1048 | TA0010 (Exfiltration) |
| Privilege escalation | T1068 | TA0004 (Privilege Escalation) |
| Account manipulation | T1098 | TA0003 (Persistence) |
| Suspicious process | T1059 | TA0002 (Execution) |
| Firewall evasion | T1205 | TA0005 (Defense Evasion) |
| Honeypot / canary | T1595 | TA0043 (Reconnaissance) |

## XQL Embedding Rules

### Literal Block Scalar

The `xql_query` field uses YAML literal block scalar syntax (`|`) to preserve line
breaks and indentation in the XQL query:

```yaml
xql_query: |
  dataset = xdr_data
  | filter event_type = ENUM.EVENT_LOG
    and action_evtlog_event_id = 4625
  | comp count() as failed_count by agent_hostname
  | filter failed_count >= 10
```

### What Goes in XQL vs YAML

| Concern | Where It Lives | Example |
|---------|---------------|---------|
| Event filtering | XQL | `filter event_type = ENUM.EVENT_LOG` |
| Field extraction | XQL | `alter user = event_data -> TargetUserName` |
| Aggregation / threshold | XQL | `comp count() ... \| filter count >= 5` |
| Alert name | YAML `alert_name` | `Failed Auth Spike` |
| Severity (static) | YAML `severity` | `SEV_030_MEDIUM` |
| Severity (dynamic) | XQL `if()` | `alter xdm.alert.severity = if(score >= 90, "CRITICAL", ...)` |
| MITRE mapping | YAML `mitre_defs` | `"T1110": ["TA0006"]` |
| Suppression | YAML `suppression_*` | `suppression_fields: [source_ip]` |

### Dynamic Severity in XQL

When the source data has a numeric score but no discrete severity field, compute
severity dynamically in XQL:

```xql
| alter xdm.alert.severity = if(
    score >= 90, "CRITICAL",
    score >= 70, "HIGH",
    score >= 50, "MEDIUM",
    "LOW"
  )
```

Always set a static fallback in the YAML `severity` field (typically `SEV_040_LOW`)
so the alert has a severity even if the `if()` chain produces no match.

## Complete Annotated Template

```yaml
# ANNOTATION: global_rule_id must be a valid UUID v4 -- generate a new one for each rule
global_rule_id: "xxxxxxxx-xxxx-4xxx-yxxx-xxxxxxxxxxxx"

# ANNOTATION: rule_id is the machine-readable identifier; use snake_case
rule_id: "vendor_detection_name"

# ANNOTATION: rule_name is the human-readable display name
rule_name: "Vendor - Detection Name"

# ANNOTATION: rule_labels are optional tags for organizing rules in the UI
rule_labels:
  - "detection_category"
  - "data_source"

# ANNOTATION: rule_source identifies who created the rule
rule_source: "xsiam-buddy"

# ANNOTATION: name and alert_name MUST be identical
name: "Vendor - Detection Name"
alert_name: "Vendor - Detection Name"

# ANNOTATION: description should explain what the rule detects and why it matters
description: >
  Detects <specific behavior> from <data source>. This indicates <why it matters>
  and should be investigated by <response guidance>.

# ANNOTATION: dataset is always "alerts" for correlation rules
dataset: alerts

# ANNOTATION: action is always "ALERTS" for correlation rules
action: ALERTS

# ANNOTATION: alert_domain classifies the alert by security domain
alert_domain: NETWORK

# ANNOTATION: alert_category maps to a MITRE tactic or functional category
alert_category: "Credential Access"

# ANNOTATION: execution_mode determines how the rule is triggered
# REAL_TIME = every matching event; SCHEDULED = on a cron schedule
execution_mode: SCHEDULED

# ANNOTATION: search_window defines the lookback period for the query
# For SCHEDULED rules, set to at least 2x the cron interval
search_window: "15m"

# ANNOTATION: crontab is required for SCHEDULED rules; omit for REAL_TIME
crontab: "*/5 * * * *"

# ANNOTATION: xql_query contains detection logic ONLY
# Alert metadata (name, severity, MITRE) lives in the YAML wrapper
xql_query: |
  dataset = xdr_data
  | filter <conditions>
  | alter <field extractions>
  | comp <aggregation> by <group fields>
  | filter <threshold>
  | fields <output columns>

# ANNOTATION: severity enum -- valid values are SEV_010_CRITICAL, SEV_020_HIGH,
# SEV_030_MEDIUM, SEV_040_LOW, SEV_060_INFORMATIONAL (no SEV_050)

# ANNOTATION: user_defined_severity allows field-based severity overrides (optional)
# user_defined_severity:
#   field_name:
#     "critical_value": SEV_010_CRITICAL
#     "high_value": SEV_020_HIGH

# ANNOTATION: user_defined_category provides a custom category in the alerts table
# user_defined_category: "Custom Category"

# ANNOTATION: suppression prevents duplicate alerts from flooding the SOC
suppression_enabled: true

# ANNOTATION: suppression_duration -- how long to suppress duplicates
suppression_duration: "1h"

# ANNOTATION: suppression_fields -- which fields define a "duplicate"
suppression_fields:
  - "source_ip"

# ANNOTATION: mitre_defs maps technique IDs to tactic IDs
mitre_defs:
  "T1110":
    - "TA0006"

# ANNOTATION: mapping_strategy defaults to XQL; use CUSTOM for manual field mapping
# mapping_strategy: XQL

# ANNOTATION: alert_fields is used with mapping_strategy: CUSTOM
# alert_fields:
#   - field_name: "custom_field"
#     source: "xql_field_name"

# ANNOTATION: drilldown_query_timeframe sets the time window for the drilldown link
# drilldown_query_timeframe: "1h"

# ANNOTATION: investigation_query_link provides a deep link for investigation
# investigation_query_link: ""
```

## Field Order Validation

When reviewing a correlation rule YAML file, verify the field order matches the
canonical order listed above. XSIAM exports use this exact order, and maintaining
consistency makes diff comparisons reliable and avoids unnecessary merge conflicts.

Fields that are omitted (optional fields not used) do not affect order validation --
only the relative order of present fields matters.
